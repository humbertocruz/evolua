# App Evolution Phases

## Ideia central

O nome **Evolu[a]** sugere que um app não nasce pronto.

Ele deve poder **evoluir por fases**, ganhando estrutura, visual, dados, comportamento e materialização ao longo do tempo.

Em vez de tratar o desenvolvimento como geração instantânea de software completo, o Evolu[a] pode tratar o app como um organismo/projeto em evolução progressiva.

## Princípio

> O papel do Evolu[a] não é apenas construir apps.
> O papel do Evolu[a] é permitir que apps **evoluam com clareza e estrutura** ao longo do tempo.

## Por que isso importa

- é mais realista
- combina com o desenvolvimento de software na prática
- reduz a expectativa de "geração mágica completa"
- encaixa melhor com a ideia de modelo multidimensional
- permite que o produto acompanhe o amadurecimento do app

## Fases candidatas

### Fase 0 — Ideia / Esqueleto
O app ainda é uma intenção estruturada.

Pode conter:
- nome do app
- páginas principais
- estrutura inicial
- objetivos
- fluxos básicos

### Fase 1 — Structure First
A organização principal do app aparece.

Pode conter:
- páginas
- seções
- componentes
- slots
- hierarquia estrutural

### Fase 2 — Visual Materialization
A estrutura ganha aparência.

Pode conter:
- layout
- props visuais
- estilos
- variantes
- identidade visual inicial

### Fase 3 — Data Layer
O app ganha dados reais e fluxo de informação.

Pode conter:
- entidades
- state
- queries
- bindings
- relações de dados

### Fase 4 — Behavior Layer
O app ganha lógica viva.

Pode conter:
- eventos
- ações
- efeitos
- navegação
- transições
- regras de comportamento

### Fase 5 — Materialization / Build
O app se torna artifact executável/materializado.

Pode conter:
- builder
- geração de código
- exportações
- artifacts
- runtime materializado

### Fase 6 — Evolution / Refinement
O app amadurece e continua evoluindo.

Pode conter:
- semantic diff
- novas projeções
- múltiplos visuais
- colaboração
- IA ajudando na evolução
- refatorações estruturais

## Leitura importante

As fases não precisam ser completamente rígidas.

Elas funcionam mais como:
- estágios de maturidade
- camadas de evolução
- marcos de clareza

Não como uma prisão burocrática.

## Relação com as dimensões

### Structure
Aparece cedo e sustenta o resto.

### Visual
Pode vir depois da estrutura.

### Data
Pode surgir quando o app deixa de ser apenas composição.

### Behavior
Pode amadurecer quando o app ganha regras e interações vivas.

### Code / Spatial
Aparecem como materializações/projeções derivadas.

## O papel do Evolu[a]

O Evolu[a] pode ajudar o dev a:
- entender em que fase o app está
- evoluir o app de uma fase para outra
- aplicar operações coerentes com o estágio atual
- não pular diretamente para complexidade desnecessária

## Possíveis consequências de produto

- templates por fase
- recomendações de evolução
- operações sugeridas por estágio
- IA ajustando a linguagem conforme a fase do app
- semantic diff mostrando mudança de maturidade

## Resumo curto

> O Evolu[a] pode ser entendido não apenas como um editor, mas como um sistema para conduzir a evolução progressiva de apps ao longo de fases de maturidade.
